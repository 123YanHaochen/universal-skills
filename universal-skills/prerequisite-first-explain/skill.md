# 知识前置讲解

## 核心理念

AI讲解时最常见的问题是：用用户不懂的概念去解释用户不懂的概念，导致讲解完全无效。

本技能的核心机制：**讲解开始前，先确保双方对"你已经会什么"达成共识。** 所有讲解中会出现的概念都必须先列出来让用户确认，用户确认理解后才开始讲解。有不理解的，先递归补齐。

## 工作流程

### Step 0: 接收学习对象

用户提供的学习对象可以是任何形式：
- 一个术语或概念（"RAII"、"TCP三次握手"、"闭包"）
- 一段代码
- 一个文件
- 一篇文档或文章
- 用户用自然语言描述的问题
- 任何可以被学会的事物

如果学习对象不明确，先向用户确认具体要讲什么。

### Step 1: 确定讲解深度

向用户展示以下选项，请用户选择（不要替用户决定）：

1. **速览级** — 用一句话或一个类比解释，知道它"是什么"即可
2. **实用级** — 理解它能做什么、什么时候用、怎么用
3. **原理级** — 理解它底层是怎么工作的，为什么这样设计
4. **专家级** — 深入所有细节，包括边界情况、设计权衡、演进历史
5. **自定义** — 由用户描述具体想要什么（比如"我只关心它在并发场景下的行为"）
6. **最低必要级** — 只学到足以支撑理解上层概念的程度，不额外深入（此选项仅在递归补齐子概念时出现，顶层学习目标不提供此选项）

等用户明确选择后才进入下一步。

### Step 2: 知识盘点

这是本技能最关键的步骤。

根据用户选择的深度，分析你要讲解的内容，**列出讲解中会涉及的所有概念、术语、前置知识**。

**输出格式：**

```markdown
## 知识盘点

要按照【XX级】讲解【学习对象】，讲解中会涉及以下概念：

### 讲解中会使用的概念
| 序号 | 概念 | 为达成目标需掌握的程度 | 原子性 |
|------|------|------------------------|--------|
| 1 | **概念A** | 只需知道它的定义和作用，不需要了解实现细节 | 🔵 复合 |
| 2 | **概念B** | 需要能自己写出示例代码的程度 | ⚫ 原子 |
| 3 | **概念C** | 只需知道它解决了什么问题，不需要了解原理 | ⚫ 原子 |
| ... | ... | ... | ... |

### 基础背景（如果以下任何一项你不熟悉，也请告诉我）
- 基础背景1
- 基础背景2
```

**为达成目标需掌握的程度——编写原则：**

不再说"这个概念用来解释 XXX"——那是备课笔记，不是学习导航。用户需要知道的是：**为了学会最终目标，这个名字最少要学到什么程度。**

确定掌握程度的唯一标准：
- 掌握得**比这个多** → 浪费时间，对学会最终目标没有帮助
- 掌握得**比这个少** → 不足以搭建通往最终目标的知识阶梯

**编写指南：**

1. **以最终目标为锚点。** 同一个概念，最终目标不同，所需掌握程度就不同。比如"哈希表"——为了学算法复杂度需要理解冲突解决，为了用 Python 字典只需要知道 key-value 存取。
2. **描述边界，不只描述内容。** 不光说"你需要懂 X"，还要说"但不需要懂 Y"。明确画线，让用户知道学到这里就够了。
3. **用具体行为动词。** "能写出""能识别""能解释""只需知道存在"——比"了解""理解"更精确。
4. **参考高考考纲的精度。** 高考考纲不会只说"掌握三角函数"，而是说"能运用三角恒等式进行恒等变换，能解三角形"。你的掌握程度描述也要达到这个精度——用户看完就知道自己差什么、补多少。

**原子性标记说明：**

- ⚫ **原子** — 此概念在当前深度下不可再分，可以直接解释，不需要引入其他前置概念。原子概念是递归的"叶子节点"，用户说不懂就直接讲解，无需再做知识盘点。
- 🔵 **复合** — 此概念由其他更基础的概念构成。如果用户不理解，需要进入下一层递归，对该概念单独做知识盘点。

**如何判断原子性：**
- 能在一段话内讲清楚、不需要引用其他专有名词的概念 → 原子
- 讲解时会自然引出其他术语、需要用户先知道别的东西才能理解的概念 → 复合
- 原子性是相对的：同一概念在"速览级"可能是原子的，在"原理级"可能变成复合的

**知识盘点的原则：**

- **宁可多列，不可漏列。** 凡是在讲解中会出现的专有名词、技术术语、理论概念，都必须列出。一个概念如果你不确定用户是否知道，列出来让用户判断。
- **不要自作主张跳过"基础"概念。** 你认为的"常识"可能就是用户的知识盲区。列表宁长勿短。
- **每个概念标注"为达成目标需掌握的程度"**，精确描述该概念最少要学到什么程度，帮用户判断自己是否达标。
- **每个概念必须标记原子性**，让用户预判哪些概念点进去还会有下一层递归、哪些是叶子节点一次性讲完。
- **分组整理**，让列表易于扫读，比如把10个概念分成2-3组。
- **基础背景单独列出**，把最底层的前置假设（比如"你对Python语法有基本了解吗？"）单独确认。

### Step 3: 确认理解

展示知识盘点后，等待用户回复。用户可能：

- **"都理解"** → 所有概念确认完毕，进入 Step 5 正式讲解
- **"X、Y、Z 不理解"** → 进入 Step 4，对每个不理解的概念递归处理
- **"有些我不确定"** → 帮用户澄清：用一句话解释不确定的概念，让用户判断是否理解

### Step 4: 递归补齐

对用户标记为不理解的概念，逐个处理：

```
对每个不懂的概念：
    → 将它作为新的"学习对象"
    → 回到 Step 1，重新确定深度
    → 执行 Step 2 知识盘点
    → 执行 Step 3 确认理解
    → 如果有更深层的不懂，继续递归
    → 只有所有子概念用户都确认理解后，才回到上一层继续
```

> **注意：** 递归进入 Step 1 时，深度选项会多出第 6 项「最低必要级」——只学到足以支撑上层概念的程度。顶层学习目标不提供此选项。

**递归深度由用户控制：**
- 用户可以随时说"到这里就行，给我一个简化版"来中断递归
- 用户可以选择对某个子概念降低深度（比如主概念学"原理级"，但某个前置概念只需要"速览级"）
- 用户也可以说"太深了，回到上一层吧"

**递归中掌握程度的传递：**

用户为子概念选择「最低必要级」时，只讲到足以支撑理解上一层概念的程度，不展开。选其他级别则按该级别的正常深度讲解。如果用户选了比"最低必要级"更深的级别，也尊重选择——用户可能有自己的学习规划。

核心约束：子概念的知识盘点中，"为达成目标需掌握的程度"始终以上一层概念为锚点，最终追溯到原始学习目标。子概念不是独立专题，而是通往上一层的台阶。

**递归树：**

每次进入递归时，维护并展示一棵 ASCII 递归树，让用户看到全局学习路径和当前位置。树的结构以原始学习目标为根，以概念间的依赖关系为父子边。

**树的构建规则：**
- 根节点 = 用户最初要学的目标
- 一级子节点 = 根节点知识盘点中列出的所有概念
- 当某个复合概念触发递归时，它的知识盘点中的概念成为其子节点
- 原子概念永远是叶子节点（标记 `⚫`），不会再生长子节点
- 用空格和 `├──` / `└──` / `│` 绘制标准 ASCII 树形

**节点状态标记：**
| 标记 | 含义 |
|------|------|
| ✅ | 已确认理解，该子树已完成 |
| 🔄 | 正在处理此节点（正在等待用户选择深度或正在讲解） |
| ⬜ | 待处理，尚未开始 |
| ⚫ | 原子概念，不可再分（叶子节点） |

**展示时机：**
- 每次进入新的递归层级时（开始讲解一个概念前），展示完整树
- 一个节点处理完毕后（用户确认理解），展示更新后的树再进入下一个节点
- 树比较大的时候（超过 20 行），只展示当前节点往上两层的局部树，加上 `... (共 N 个节点，X 个已完成)` 的汇总

**树格式示例：**

```
📚 知识递归树 — 已完成 3/7 个节点

RAII (原理级)
├── ✅ 构造函数 & 析构函数 (⚫)
├── ✅ 栈上对象 vs 堆上对象 (⚫)
├── 🔄 异常安全 (最低必要级) ← 当前位置
│   ├── ✅ 异常基本概念 (⚫)
│   ├── ⬜ 异常安全保证等级 (🔵)
│   └── ⬜ noexcept说明符 (⚫)
├── ⬜ 智能指针 unique_ptr (🔵)
├── ⬜ 作用域 (⚫)
└── ⬜ 资源管理 (⚫)
```

### Step 5: 正式讲解

**只有当知识盘点中的所有概念用户都已确认理解后，才开始讲解。**

此时你可以：
- 自由使用知识盘点中已确认的概念
- 按照用户选择的深度进行讲解
- 如果在讲解过程中自然引出了新的、知识盘点中未列出的概念，暂停并补充到盘点中让用户确认

**讲解风格：**

- **深入浅出**。把复杂原理拆解成直觉能理解的语言，用简单的词汇讲清楚复杂的事。
- **通俗易懂**。避免堆砌术语，能用大白话说清楚的就别用术语。如果必须用术语，先确保该术语已在知识盘点中被用户确认过。
- **指代清晰**。不用"它""这个""那个"等模糊指代，尤其是在跨句引用时。被指代的对象要明确写出来。
- **类比要回归概念本身**。类比是帮用户建立直觉的桥梁，不是终点。每用一个类比之后，必须把类比映射回概念本身——明确说出类比中的哪个部分对应概念的哪个要素、哪些细节被类比省略了。不能让用户只记住类比而没学会概念。

**讲解结束后的收尾：**
- 问一句"还有什么地方不清楚吗？"
- 如果用户想深入了解某个方面，回到 Step 1 重新确定深度

## 关键原则

### 1. 永远不要假设用户知道什么
一个概念只要出现在你的讲解中，就应该在知识盘点中列出。你眼里的"基础"可能正是用户的知识缺口。

### 2. 知识盘点要力求完整
这是整个机制的核心——如果盘点有遗漏，讲解中就会出现用户不懂的概念，整个流程就白费了。列出概念时，想象自己正在写讲解稿，然后扫描稿中每个术语是否都在盘点中。

### 3. 用户是知识水平的最终裁判
不要替用户判断"这个概念你应该会吧"。把概念列表交给用户，让他自己判断。用户最清楚自己懂什么不懂什么。

### 4. 递归深度用户说了算
你负责列出前置概念和讲解，用户负责决定钻多深、在哪里停。不要替用户终止递归，也不要强迫用户往下钻。

### 5. 耐心，这个过程需要时间
这个流程比直接讲解更耗时，但它确保用户真正学会。不要为了快而跳过步骤。

## 常见边界情况

- **用户说"我什么都不懂"**：从最基础的知识盘点开始，列出所有概念让用户识别哪些不懂
- **用户说"基础知识我都懂"**：仍然做知识盘点，但可以把基础背景省略，只列出领域特定的概念
- **用户想跳过知识盘点直接听讲解**：温和解释知识盘点的目的——"为了确保我能用你听得懂的语言讲，让我先确认一下你的知识基础。如果讲到一半发现术语你不懂，反而更浪费时间。"如果用户坚持跳过，尊重选择，但提醒这可能影响讲解效果
- **学习对象极其简单**：知识盘点自然会很短，流程依然有效，不需要特殊处理
- **学习对象极其复杂**：知识盘点本身就是一种学习导航，把复杂主题拆解成概念清单已经帮助用户建立了框架
- **用户的回答模棱两可**（如"还行""大概懂"）：追问一个具体问题来验证，比如"你能用一句话描述析构函数是做什么的吗？"来帮助用户自我判断
